Kubernetes Pods 101
A pod is the smallest unit in a Kubernetes cluster. A pod may contain one or more containers. Pods can be scheduled (in Kubernetes terminology, scheduling means deploying) to a node.
When And Why Host More Than One Container In A Pod?
Creating a Pod
apiVersion: v1 kind: Pod metadata: name: webserver spec: containers: - name: webserver image: nginx:latest ports: - containerPort: 80
YAML
using k8sCore = Pulumi.Kubernetes.Core.V1; var webserver = new k8sCore.Pod("webserver", new PodArgs { Metadata = new ObjectMetaArgs { Name = "webserver" }, Spec = new PodSpecArgs { Containers = new ContainerArgs { Name = "webserver", Image = "nginx:latest", Ports = new ContainerPortArgs { ContainerPortValue = 80 } } } });
C#
Viewing Your Pods
kubectl get pods NAME READY STATUS RESTARTS AGE webserver 1/1 Running 0 11m
Bash
Which Node Is This Pod Running On?
kubectl get pods -o wide NAME READY STATUS RESTARTS AGE IP NODE NOMINATED NODE READINESS GATES webserver 1/1 Running 0 11m 10.1.0.247 docker-desktop <none> <none>

I Need The Output In ...
kubectl get pods -o YAML
Bash
Like other Kubernetes components, kubectl was designed to be modular. You can have its output piped to other tools. Commonly, tools use JSON as their preferred communication language. Kubectl allows you to have the output of its subcommands in either YAML or JSON, with YAML being the default.
You can easily pipe it to a JSON parser like jq:
kubectl get pods -o json | jq
Bash
Let’s Add A Second Container To The Pod
apiVersion: v1 kind: Pod metadata: name: webserver spec: containers: - name: webserver image: nginx:latest ports: - containerPort: 80 - name: webwatcher image: afakharany/watcher:latest
YAML
var webserver = new k8sCore.Pod("webserver", new PodArgs { Metadata = new ObjectMetaArgs { Name = "webserver" }, Spec = new PodSpecArgs { Containers = new ContainerArgs[] { new ContainerArgs { Name = "webserver", Image = "nginx:latest", Ports = new ContainerPortArgs { ContainerPortValue = 80 } }, new ContainerArgs { Name = "webwatcher", Image = "afakharany/watcher:latest" } } } });
C#
Containers = new List<ContainerArgs> { new ContainerArgs { Name = "webserver", Image = "nginx:latest", Ports = new ContainerPortArgs { ContainerPortValue = 80 } }, new ContainerArgs { Name = "webwatcher", Image = "afakharany/watcher:latest" }
C#
The only change we made here is adding a new container to the containers array. The name of the container now is more important than before; as it will be used whenever you need to execute commands against this specific container in the pod.
Notice that the READY column now has two containers running out of two.
kubectl get pods NAME READY STATUS RESTARTS AGE webserver 2/2 Running 0 12m
Bash
Communicating with localhost provides as low network latency as possible.
The Pod Is Failing But I Don’t Know Why
When you need more details about a pod. kubectl describe pods will give you more than just the status of the image. It will print any logs that the pod produces, which is invaluable when troubleshooting a misbehaving pod.
kubectl describe pods webserver
Bash
Basic logging in Kubernetes
Everything a containerized application writes to stdout and stderr is handled and redirected somewhere by a container engine. For example, the Docker container engine redirects those two streams to a logging driver, which is configured in Kubernetes to write to a file in json format.
You can see the outputs data to the standard output stream of containers that writes it. To fetch the logs, use the kubectl logs command:
kubectl logs webserver
Bash
If your pod has multiple containers, you should specify which container's logs you want to access by appending a container name to the command:
kubectl logs webserver -c webserver
Bash
To follow logs of a pod/container use -f flag.
By default, if a container restarts, the kubelet keeps one terminated container with its logs, to see this logs use the -p flag. If a pod is evicted from the node, all corresponding containers are also evicted, along with their logs.
Executing Commands Against Pods
If you’d probably needed to execute commands against running containers, mostly for troubleshooting purposes. Kubectl has borrowed this functionality from Docker. You can issue a very similar command against the pod as follows:
kubectl exec -it webserver -- /bin/bash
Bash
A pod can contain multiple containers, so which container will kubectl execute bash against? Good question. If you want to run the command against a specific container in a multi-container pod, you can do this by adding -c container name option.
If your pod has multiple containers, you should specify which container's you want to access by appending a container name to the command:
kubectl exec -it webserver -c webwatcher -- /bin/bash
Bash
Notice that we used the pod name not the container. Remember, pods host containers. We also used the -- to inform kubectl that we’re no longer adding any more subcommands or arguments and that the coming is the actual command that needs to be executed against the pod.
Terminating a Pod
By default, Kubernetes will restart a pod if it exits. Let’s see this in action:
kubectl exec -it webserver -- /bin/bash -c "kill 1"$
Bash
But what if we really need the pod to go down? The following command will do the job:
kubectl delete pods webserver
Bash
